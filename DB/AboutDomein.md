以下が修正済みドキュメントです。

ドメイン単位のDB設計見直しチェックリスト
ドメインごとにDB設計を見直すときは、単に性能改善だけではなく、データ構造・利用パターン・運用要件まで含めて洗い出すことが重要です。MySQL / PostgreSQL を前提に、確認すべき主要ポイントを整理します。

1. ビジネス要件・利用パターンの再確認
	∙	そのドメイン（機能）がどのようなデータを持ち、どの頻度で更新/参照されるかを把握する
	∙	参照主体か、書き込み主体かを判断する
	∙	集計・検索条件が固定的か、動的に変わるかを確認する
	∙	将来のデータ増加を予測する（1年後・3年後の想定件数）

2. 正規化と非正規化のバランス
正規化
	∙	冗長性を排除して整合性を保つ
	∙	更新系（INSERT/UPDATE/DELETE）が多いシステムでは特に重要
非正規化
	∙	JOIN回数削減や集計性能向上のために冗長データを持つ
	∙	レポート・分析系やキャッシュテーブルで活用
設計判断として、ドメインごとにトランザクション系（OLTP）か分析系（OLAP）かを切り分けることが前提になります。

3. インデックス戦略
クラスタ化インデックスの挙動の違い
	∙	MySQL（InnoDB）: 主キーがクラスタ化インデックスとなり、データページと一体で物理格納される。主キー選択がアクセス性能に直結する
	∙	PostgreSQL: デフォルトはヒープ構造（非クラスタ化）。CLUSTERコマンドで物理順序を変更できるが、以降の更新では維持されないため運用上の注意が必要
複合インデックス
	∙	カラムの順序はクエリの絞り込み条件・ソート順に合わせる（左端一致の原則）
部分インデックス・関数インデックス
	∙	PostgreSQL: WHERE句付きの部分インデックスをネイティブサポート。特定条件のクエリに絞った効率的なインデックスが作れる
	∙	MySQL 8.0以降: 関数インデックス（式インデックス）をサポート。WHERE句による部分インデックスはサポートしていない
不要インデックスの整理
	∙	インデックスが多いと更新系のオーバーヘッドが増加するため、未使用インデックスは定期的に棚卸しする

4. パーティショニングとアーカイブ
データの寿命・アクセス頻度を考慮して設計します。
パーティション方式
	∙	MySQL: RANGE / LIST / HASH / KEY パーティションをサポート
	∙	PostgreSQL 10以降: 宣言的パーティショニング（RANGE / LIST / HASH）をネイティブサポート。pg_partmanなどの拡張と組み合わせると運用が楽になる
アーカイブ戦略
	∙	参照頻度の低い古いデータを別DBやオブジェクトストレージに移動し、本番の負荷を軽減する
	∙	時系列データはパーティション分割が有効

5. クエリの最適化
	∙	実行計画の確認
	∙	MySQL 8.0以降: EXPLAIN ANALYZEが利用可能。それ以前はEXPLAIN FORMAT=JSONで代替
	∙	PostgreSQL: EXPLAIN ANALYZEで実際の実行時間・行数を確認できる
	∙	SELECT *禁止 → 必要なカラムのみ取得する
	∙	ページネーションの見直し: OFFSET方式は件数が増えるほど遅くなるため、seek method（カーソルページネーション）への移行を検討する
	∙	JOINを減らす、または集約テーブルを活用する

6. スキーマ変更のしやすさ
	∙	将来の項目追加を見越して、型やNULL可否を設計段階で考慮する
	∙	ENUM型よりマスタテーブル参照を優先する（値の追加・変更が柔軟になる）
	∙	大規模テーブルのALTER TABLEは本番停止リスクがあるため、次の手段で対処する
	∙	MySQL: オンラインDDL（InnoDB）、またはpt-online-schema-change / gh-ostを活用
	∙	PostgreSQL: CREATE INDEX CONCURRENTLYなどの並行操作、またはpg_repackを活用

7. 運用監視・メンテナンス性
監視項目
	∙	クエリ遅延（スロークエリログ）
	∙	接続数・コネクションプールの枯渇
	∙	ロック競合（information_schema.INNODB_LOCKS / pg_locks）
	∙	ストレージ使用量の増加傾向
定期メンテナンス
	∙	PostgreSQL: VACUUM / ANALYZEの自動実行設定を確認する（autovacuumの設定値を本番負荷に合わせてチューニング）
	∙	MySQL: OPTIMIZE TABLEやANALYZE TABLEを定期実行する
バックアップとリカバリ
	∙	バックアップとリストアの手順を定期的に検証する
	∙	障害時のリカバリ時間（RTO）とデータ損失許容度（RPO）をドメインごとに定義する

8. セキュリティとアクセス権限
	∙	ドメイン単位でDBユーザと権限を分離し、最小権限の原則を徹底する
	∙	SQLインジェクション対策としてプリペアドステートメントを使用する（ORMを経由する場合も素のクエリを直接組み立てていないか確認する）
	∙	個人情報や機密データには暗号化を適用する
	∙	カラム単位の暗号化（アプリ層またはDB機能）
	∙	ディスク暗号化（TDE: Transparent Data Encryption）
	∙	通信経路の暗号化（SSL/TLS）

修正の要点は「MySQLとPostgreSQLの挙動の違いを混同せずに分けた」「EXPLAIN ANALYZEのバージョン条件を正確にした」「ドキュメント外の文言を削除した」の3点です。表形式への変換が必要な場合は言ってください。​​​​​​​​​​​​​​​​
